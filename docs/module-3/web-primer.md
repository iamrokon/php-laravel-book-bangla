# Episode 1-5: Web Primer

## The Internet and The Web
কম্পিউটার আবিষ্কারের শুরুর দিকে একটি কম্পিউটার নিজে নিজেই কাজ করত। এরপর প্রয়োজন দেখা দেয় একাধিক কম্পিউটারের মধ্যে তথ্য আদান-প্রদান করার। এভাবে অনেকগুলো কম্পিউটার তার দিয়ে বা ওয়্যারলেস মাধ্যমে একে অপরের সাথে কানেক্টেড হয়ে তৈরি করে নেটওয়ার্কিংয়ের। পুরো পৃথিবীর এরকম অসংখ্য নেটওয়ার্ক একত্রে সংযুক্ত (Interconnected) হয়েই তৈরি হয়েছে **Internet**। আমরা সাধারণত কোনো না কোনো ISP (Internet Service Provider) এর মাধ্যমে ইন্টারনেট কানেক্টেড থাকি।

ইন্টারনেটের উপর ভিত্তি করে গড়ে ওঠা একটি সার্ভিস বা অ্যাপ্লিকেশন হলো **Web Service** (যেমন- ওয়েবসাইট)। এছাড়া ইন্টারনেটের আরো সার্ভিসের মধ্যে রয়েছে ইমেইল, IRC (Internet Relay Chat) ইত্যাদি। অর্থাৎ, এগুলো ইন্টারনেট নয়, বরং ইন্টারনেটের উপর ভিত্তি করে তৈরি এক একটি সার্ভিস।

ডোমেইন নেম (যেমন `www.google.com`) দিয়ে ইন্টারনেটে কোনো ওয়েবসাইট ভিজিট করলে, সেই নামের আড়ালে থাকা নির্দিষ্ট সার্ভারের আইপি (IP address) বের করে দেয় **DNS (Domain Name Server)**। 

## Client Server Architecture
নেটওয়ার্কের এমন একটি ডিভাইস যার কাছে কোনো রিকোয়েস্ট (Request) পাঠালে, সে সেটি বুঝে কাঙ্ক্ষিত রেসপন্স (Response) ব্যাক করে, তাকেই সার্ভার (Server) বলা হয়। এটি মূলত ক্লায়েন্ট ও সার্ভারের কমিউনিকেশন মডেল। 

### 3-Tier Architecture
ক্লায়েন্ট-সার্ভার আর্কিটেকচারের একটি পরিচিত রূপ হলো 3-Tier Architecture, যার ৩টি প্রধান অংশ থাকে:
1. **Presentation Layer (Client Part):** ইউজার ইন্টারফেস। (যেমন: Browser, Mobile App)
2. **Business/Logic Layer (Server Part):** সার্ভার সাইডে রিকোয়েস্ট প্রসেস হয়। (যেমন: PHP অ্যাপ্লিকেশন)
3. **Data Storage (Database Part):** ডেটা সেভ করে রাখা হয়। (যেমন: MySQL/Relational DB)

বড় সিস্টেমে অনেক সার্ভার বা ডাটাবেজ থাকতে পারে। এর উপর ভিত্তি করে আরো অনেক লেয়ার বা n-Tier আর্কিটেকচার তৈরি হয়।

**Peer to Peer (P2P) Model:** ক্লায়েন্ট সার্ভারের বাইরেও এই মডেল আছে, যেখানে প্রত্যেকটি ডিভাইস নিজেই ক্লায়েন্ট, নিজেই সার্ভার হিসেবে কাজ করে (যেমন: ব্লকচেইন, টরেন্ট)।

## Components of a Web Application
একটি ওয়েবসাইটের মূলত দুটি সাইড থাকে-
1. **Frontend (Client-side):** এতে থাকে HTML (স্ট্রাকচার), CSS (স্টাইল) এবং JS (ইন্টার্যাকশন বা ব্রাউজার সাইড লজিক)।
2. **Backend (Server-side):** 
   - **Web Server:** (যেমন- Apache, Nginx) এটি মূলত HTTP রিকোয়েস্ট এবং স্ট্যাটিক ফাইল হ্যান্ডেল করে।
   - **Application Server:** (যেমন- PHP/Laravel, Node.js) ডাইনামিক রিকোয়েস্ট বা বিজনেস লজিক হ্যান্ডেল করে।
   - **Database Server:** ডেটা সংরক্ষণের জন্য।
   - **Cache Server:** ডাটাবেজ থেকে বারবার ডেটা রিড করার পরিবর্তে ফাস্ট রেসপন্স করতে।

## HTTP Basics
**HTTP** এর পূর্ণরূপ Hyper Text Transfer Protocol। প্রটোকল হলো এমন কিছু রুল বা গাইডলাইন, যার উপর ভিত্তি করে দুইটি ডিভাইস (যেমন ক্লায়েন্ট ও সার্ভার) নিজেদের মধ্যে কমিউনিকেশন করে এবং ডেটার ফরম্যাট কী হবে তা ডিসাইড করে।

### HTTP Request Format 
ক্লায়েন্টের পাঠানো রিকোয়েস্ট একটি নির্দিষ্ট ফরম্যাটে থাকে- 
```http
GET /books HTTP/1.1
Host: developer.mozilla.org
Accept-Language: fr
```
এখানে প্রথম লাইনে থাকে Method (GET), Path (/books), এবং Protocol Version (HTTP/1.1)। নিচের দিকে থাকে রিকোয়েস্ট হেডার্স (Headers)।

**Common HTTP Methods:**
- `GET`: ডাটা রিড বা অ্যাক্সেস করা।
- `POST`: নতুন রিসোর্স বা ডেটা তৈরি করা।
- `PUT / PATCH`: পুরাতন রিসোর্স মডিফাই বা আপডেট করা।
- `DELETE`: কোনো রিসোর্স ডিলিট করা।

### HTTP Response Format
সার্ভার রিকোয়েস্ট পাওয়ার পর একটি নির্দিষ্ট ফরম্যাটে রেসপন্স দেয়:
```http
HTTP/1.1 200 OK
Date: Sat, 09 Oct 2010 14:28:02 GMT
Server: Apache
Content-Type: text/html

(Response Body content here...)
```
এখানে প্রথম লাইনে থাকে Status Code (200) এবং Status Message (OK)।

**HTTP Status Codes:**
- `2xx`: Success (200 OK, 201 Created)।
- `3xx`: Redirection (301 Moved Permanently)। 
- `4xx`: Client Error (404 Not Found, 403 Forbidden)। রিকোয়েস্টের ফরম্যাট বা ডেটাতে ভুল থাকলে।
- `5xx`: Server Error (500 Internal Server Error, 502 Bad Gateway)। সার্ভারের সাইডে কোনো এরর (Error) হলে।

## Cookie and Session
HTTP প্রটোকল হলো **Stateless**। এর মানে হলো, প্রতিবার রিকোয়েস্ট পাঠানোর পর সার্ভার যখন রেসপন্স দেয়, তখন সে পূর্বের কোনো স্টেট (State) বা ইনফরমেশন মনে রাখে না। 

কিন্তু লগইন সিস্টেম বা ইউজারের স্টেট ম্যানেজমেন্টের ক্ষেত্রে এটা মনে রাখা প্রয়োজন। এজন্য **Session** ব্যবহার করা হয়।
1. ইউজার লগইন করলে সার্ভার একটি ইউনিক `SESSION ID` জেনারেট করে এবং সেটি ডেটাবেজ বা ফাইলে ইউজার ইনফরমেশনের সাথে সেভ করে রাখে।
2. এরপর রেসপন্সের সাথে ঐ `SESSION_ID`-টি ক্লায়েন্ট বা ব্রাউজারকে পাঠিয়ে দেয়।
3. ব্রাউজার এই আইডিটিকে তার **Cookie**-তে স্টোর করে।
4. এরপর ব্রাউজার প্রতিবার রিকোয়েস্ট করার সময় কুকি থেকে ঐ সেশন আইডিটি সাথে পাঠিয়ে দেয়।
5. সার্ভার সেটি ম্যাচ করে নির্দিষ্ট ইউজারকে সনাক্ত করে রেসপন্স জেনারেট করে।

কুকির একটি এক্সপায়রেশন (Expiration) ডেট থাকে, ঐ সময় পার হলে কুকি ডিলিট হয়ে যায়। "Remember me" অপশন ক্লিক করলে কুকির এক্সপায়রেশন বাড়িয়ে দেওয়া হয়।
